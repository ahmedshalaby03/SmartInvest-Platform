# تصميم: إسناد المشروعات الفرعية للجهات التنفيذية وتعيين المقاولين

**التاريخ:** 2026-07-22
**الحالة:** معتمد (Approved) — جاهز لكتابة خطة التنفيذ
**النطاق:** Backend فقط. الواجهة الأمامية (Angular) تُبحث لاحقًا في محادثة/سبيسفكيشن منفصلة.

## السياق

هذا هو **سب-بروجكت 1** من طلب أكبر يتضمن أيضًا سب-بروجكت 2 (مسار متابعة المشروع: الجدول الزمني، الصور، الملاحظات، المرفقات، نسبة الإنجاز، نسبة الصرف). سب-بروجكت 2 يعتمد على هذا التصميم (يحتاج حسابات الجهات والمقاولين موجودة أولًا) وسيُبحث في سبيسفكيشن منفصل لاحقًا.

النظام الحالي (SmartInvest) نظام تخطيط استثماري لمحافظة المنوفية، بواجهة عربية RTL، بنية Onion (.NET 10)، دورين حاليين فقط: `PlanningManager` و`PlanningEmployee`.

## نظرة عامة على الحل

1. كل مشروع فرعي (`SubProject`) يُسند لجهة تنفيذية واحدة (`ExecutiveAgency`).
2. الجهة التنفيذية لها حساب دخول واحد (1:1)، وتقدر تعيّن مقاول واحد أو أكثر على المشروعات المسندة ليها، كل تعيين بنوع عقد (`ContractType`) محدد.
3. كل مقاول له حساب دخول واحد (1:1) أيضًا.
4. مدير التخطيط له صلاحية كاملة (تجاوز) على كل عمليات التخطيط، بلا استثناء — مبدأ عام يُطبّق في كل قواعد الصلاحيات أدناه وفي أي مرحلة قادمة.

## نموذج البيانات

### تعديلات على كيانات موجودة

**`SubProject`**
- إضافة `ExecutiveAgencyId` (`int?`, FK إلى `ExecutiveAgency`, nullable حتى يتم الإسناد).

**`ProjectAssignment`** (الكيان موجود بالفعل بدون أي طبقة API فوقه)
- **حذف** `ExecutiveAgencyId` منه بالكامل — الجهة المالكة تُشتق دائمًا من `SubProject.ExecutiveAgencyId` وقت القراءة، لتفادي احتمال تعارض بين جهة التعيين وجهة المشروع.
- **إضافة** `IsLocked` (`bool`, default `false`) — انظر "قاعدة القفل عند تغيير الجهة" أدناه.
- باقي الحقول كما هي: `ContractorId` (nullable), `ContractTypeId`, `AssignmentDate`, `ContractNumber`, `ContractValue`, `ExpectedStartDate`, `ExpectedEndDate`, `Notes`.

**`ApplicationUser`**
- إضافة `ExecutiveAgencyId` (`int?`, nullable, FK إلى `ExecutiveAgency`, unique).
- إضافة `ContractorId` (`int?`, nullable, FK إلى `Contractor`, unique).
- قاعدة عمل: مستخدم واحد بحد أقصى واحدة من القيمتين مربوطة (أو ولا واحدة لو موظف تخطيط عادي).

### كيانات جديدة

**`ProjectAssignmentChangeRequest`**
يمثّل طلب تعديل على `ContractValue` و/أو `ExpectedEndDate` لتعيين موجود.

| الحقل | النوع | ملاحظات |
|---|---|---|
| `ChangeRequestId` | `int` | PK |
| `AssignmentId` | `int` | FK → `ProjectAssignment` |

| `RequestedExpectedEndDate` | `DateTime?` | التاريخ الجديد المطلوب (لو مطلوب تغييره) |

قاعدة عمل: طلب `Pending` واحد بحد أقصى لكل `AssignmentId` في نفس الوقت — لازم يتحسم (Approved/Rejected) قبل تقديم طلب جديد.

### كيانات بدون تغيير هيكلي (تحتاج API فقط)
`ExecutiveAgency`, `Contractor`, `ContractType` — الكيانات موجودة بالفعل بحقولها الحالية، تحتاج فقط طبقة Repository/Service/Controller.

## الأدوار والهوية

- دوران جدد: `ExecutiveAgency`, `Contractor` (يُضافا لمصفوفة seeding الأدوار في `Program.cs` بجانب `PlanningEmployee`/`PlanningManager`).
- إنشاء حساب جهة/مقاول = إنشاء السجل (`ExecutiveAgency`/`Contractor`) + حساب الدخول (`ApplicationUser` مربوط بـ FK) **في طلب واحد**، بنفس نمط `UsersController.CreateEmployeeDto` الحالي.
- إنشاء حسابات الجهات/المقاولين: **مدير التخطيط فقط** (نفس تقييد `UsersController` الحالي `[Authorize(Roles = Roles.PlanningManager)]`).

## قواعد الصلاحيات لكل عملية

مبدأ عام يسري على كل الجدول: **مدير التخطيط له تجاوز كامل في كل عملية** بغض النظر عمّا هو مذكور.

| العملية | مين يقدر |
|---|---|
| إنشاء/تعديل جهة تنفيذية أو مقاول (+ حساب الدخول) | مدير التخطيط فقط |
| حذف جهة تنفيذية أو مقاول | مدير التخطيط فقط |
| إنشاء/تعديل نوع عقد (`ContractType`) | مدير التخطيط + موظف التخطيط |
| حذف نوع عقد | مدير التخطيط فقط |
| إسناد/تغيير الجهة التنفيذية لمشروع فرعي (`SubProject.ExecutiveAgencyId`) | مدير التخطيط + موظف التخطيط |
| تعيين مقاول لمشروع فرعي (`ProjectAssignment` - إنشاء) | مدير التخطيط + موظف التخطيط + **الجهة التنفيذية المسندة لنفس المشروع فقط** |
| تعديل بيانات التعيين العامة (`ContractNumber`, `Notes`, `ContractorId`, `ContractTypeId`) | نفس صلاحيات الإنشاء أعلاه |
| حذف تعيين مقاول | مدير التخطيط فقط |
| تعديل `ContractValue` / `ExpectedEndDate` مباشرة | مدير التخطيط + الجهة التنفيذية المسندة (تطبيق فوري) |
| طلب تعديل `ContractValue` / `ExpectedEndDate` (يحتاج موافقة) | المقاول المسند لنفس التعيين فقط |
| الموافقة/الرفض على طلب تعديل من مقاول | مدير التخطيط + موظف التخطيط + الجهة التنفيذية المسندة |

كل "نطاق ملكية" (الجهة تشوف بس مشروعاتها، المقاول يشوف بس تعييناته) يُفحص عبر مقارنة `ExecutiveAgencyId`/`ContractorId` بتاع المستخدم الحالي بالسجل المطلوب الوصول له.

## مسار طلب الموافقة (Change Request Workflow)

1. المقاول يبعث طلب تعديل (`ProjectAssignmentChangeRequest`, `Status = Pending`) — القيم الفعلية في `ProjectAssignment` **لا تتغير** لحد الموافقة.
2. أي حد من (مدير/موظف/الجهة المسندة) يقدر يوافق أو يرفض.
3. عند الموافقة: القيم تتحدّث فعليًا في `ProjectAssignment`، ويتسجّل التغيير في `AuditLog` (القيمة القديمة → الجديدة، مين طلب، مين وافق).
4. عند الرفض: الطلب يتقفل بحالة `Rejected`، بدون أي تغيير على `ProjectAssignment`.
5. لما مدير/موظف/الجهة يعدّلوا القيمتين دول مباشرة (مش عن طريق طلب من مقاول): يتسجّل الطلب تلقائيًا بحالة `Approved` (نفس الشخص طالب ومراجع) لتوحيد مسار الـ Audit، ويُطبّق فورًا.
6. لا يجوز تقديم طلب جديد لنفس التعيين طالما فيه طلب `Pending` قائم.

## قاعدة القفل عند تغيير الجهة التنفيذية للمشروع

بما إن مشروع فرعي واحد له جهة تنفيذية واحدة بس في كل لحظة، فأي `ProjectAssignment` موجود وقت تغيير `SubProject.ExecutiveAgencyId` كان بالضرورة مُنشأ تحت الجهة القديمة. لذلك:

- عند تغيير `SubProject.ExecutiveAgencyId`، كل صفوف `ProjectAssignment` المرتبطة بهذا المشروع الفرعي (في نفس الـ transaction) تُصبح `IsLocked = true`.
- التعيينات المقفولة (`IsLocked = true`) تصير للقراءة فقط لأي حد ما عدا مدير التخطيط (تجاوز كامل). ما فيش طلبات تعديل جديدة (`ProjectAssignmentChangeRequest`) تُقبل عليها.
- الجهة الجديدة تبدأ تعيينات جديدة (غير مقفولة) من الصفر على نفس المشروع الفرعي.
- التعيينات القديمة المقفولة تفضل مرتبطة بالمشروع الفرعي كأرشيف تاريخي (لا تُحذف تلقائيًا).

## معمارية الصلاحيات (فين تتحط كل فحوصة)

- **`[Authorize(Roles = "...")]`** على مستوى الـ Controller/Action: بوابة أولى، بتحدد مين يقدر يوصل لل-endpoint أصلاً بناءً على الدور بس (مش الملكية).
- **فحوصات الملكية/النطاق** (هل الجهة الحالية هي صاحبة المشروع؟ هل المقاول الحالي هو صاحب التعيين؟): تُنفّذ **داخل طبقة الـ Service (Application layer)**، مش الـ Controller — لأن نفس الـ endpoint بيُستخدم من أكتر من دور بقواعد نطاق مختلفة، والـ Controller في المشروع ده مصمم يفضل "غبي" (thin) بدون منطق أعمال، اتساقًا مع باقي الكود الحالي (`SubProjectService`, `MainProjectService`, إلخ).
- مدير التخطيط يتجاوز كل فحوصات الملكية دي تلقائيًا في طبقة الـ Service.
- هذا النمط (بوابة دور على الـ Controller + فحص ملكية داخل الـ Service) هو نفسه اللي سيُتّبع في سب-بروجكت 2 (مسار المتابعة).
